# 「Offer 捕手」项目总结报告

## 一、问题诊断

学生求职场景存在两个核心痛点：**岗位筛选成本高**和**简历命中率不确定**。

- 学生需在多个招聘平台间反复搜索，面对大量抽象 JD（如"业务敏感度强"），难以判断自身经历是否匹配，容易错过高匹配岗位。
- 通用简历建议无法对应具体岗位要求，学生不知道简历是否覆盖了关键词和能力项，导致初筛通过率不稳定。
- 求职过程缺少沉淀，投递记录、简历版本分散保存，难以复盘优化。

传统招聘平台以岗位发布与关键词搜索为核心，无法理解学生的能力结构与职业兴趣，也无法解释"为什么这个岗位适合我"。需要一个 AI 驱动的决策支持工具来弥合这一空白。

## 二、方案设计

系统定位为学生求职场景下的 **AI 匹配与简历优化助手**，采用前后端分离的分层架构，围绕以下闭环设计：

1. **简历解析**：上传简历后自动提取结构化信息（技能、经历、教育背景）。
2. **岗位理解**：导入 JD 后将其解析为职责、要求、关键词等结构化结果。
3. **匹配评分**：结合规则评分、关键词覆盖、Embedding 相似度和 LLM 评估，生成多维度可解释评分（非黑盒分数）。
4. **分层推荐**：按机会梯度生成岗位推荐组合，展示匹配度、成功率预测和风险等级。
5. **简历优化与改写**：面向具体 JD 生成可执行建议，用户确认后由 AI 生成改写草稿，展示差异预览后保存为新版本。

后端采用**模块化单体**架构（Modular Monolith），按领域纵切为独立模块，模块间仅经 service 接口协作，预留了未来拆分为微服务的边界。前端采用工作台式布局，AI 助手贯穿全流程而非仅提供聊天入口。

## 三、AI 工具选型理由

| 决策 | 选型 | 理由 |
| --- | --- | --- |
| LLM 接入 | Provider 适配层（`LLMProvider` 抽象基类） | 支持 OpenAI 兼容 API / DeepSeek / 通义等热切换，避免厂商绑定 |
| 开发测试 | Fake Provider（确定性 Mock） | 默认 `AI_PROVIDER=fake`，无需外部 API Key 即可端到端跑通全部测试和 CI |
| 向量检索 | pgvector（PostgreSQL 扩展） | MVP 阶段减少独立向量库运维成本，与主库统一管理 |
| 匹配评分 | 规则引擎 + LLM 混合 | 技能重叠、关键词覆盖等可确定性计算的维度用规则引擎保证可复现，语义理解部分交由 LLM 补充 |
| Embedding | 统一 256 维向量 | 通过 `LLMProvider.embed()` 接口标准化，模型可替换 |

这一设计确保了：测试不依赖外部服务、供应商可随时切换、评分结果可解释且部分可复现。

## 四、关键配置

- **仓库结构**：Monorepo（`apps/web` 前端 + `services/api` 后端 + `infra` 部署 + `docs` 文档），原子提交、契约共享。
- **后端**：Python 3.12 + FastAPI + SQLAlchemy 2 (async) + Alembic + PostgreSQL/pgvector + Redis + Arq（异步任务）。
- **前端**：React 18 + Vite 6 + TypeScript + TanStack Query + Tailwind CSS + shadcn/ui 风格组件。
- **环境隔离**：开发机 conda (`offerpilot`) + nvm (Node 22)；部署机 Docker + compose。`make setup` 一键搭建。
- **API 契约**：统一响应信封 `{data, requestId, timestamp}` + 统一错误码；前后端一致、可观测。
- **部署**：前端静态资源部署到 GitHub Pages / Nginx，后端 Docker 镜像独立部署。
