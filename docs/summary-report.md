# 「Offer 捕手」项目总结报告

## 一、问题诊断

学生在求职过程中面临两个核心困境。其一是岗位筛选成本极高：学生需要在多个招聘平台、学校就业网和企业官网之间反复搜索，而大量岗位描述使用"业务敏感度强""具备数据分析能力"等抽象表述，学生很难判断自己的经历是否真正匹配，极易在海量信息中错过高匹配机会。其二是简历命中率充满不确定性：学生不清楚自己的简历是否覆盖了岗位所需的关键词和能力项，市面上的通用简历建议又无法针对具体岗位给出修改方向，导致初筛通过率波动很大。与此同时，学生的投递记录、岗位偏好和简历版本往往分散保存，缺乏持续复盘和优化的基础。传统招聘平台以岗位发布和关键词搜索为核心，无法真正理解学生的能力结构与职业兴趣，也无法回答"为什么这个岗位适合我"这一关键问题。因此，我们需要一个 AI 驱动的决策支持工具来弥合这一空白。

## 二、方案设计

我们将系统定位为学生求职场景下的 AI 匹配与简历优化助手，采用前后端分离的分层架构。整个产品围绕五个核心闭环展开：系统在学生上传简历后自动提取技能、经历和教育背景等结构化信息；当学生导入岗位 JD 后，系统将其解析为职责、要求、关键词等结构化结果；随后，系统结合规则评分、关键词覆盖率、Embedding 语义相似度和大语言模型评估，生成多维度、可解释的匹配评分，而非一个不透明的黑盒分数；在此基础上，系统按机会梯度生成分层岗位推荐组合，同时展示匹配度、申请成功率预测和风险等级，帮助学生形成合理的投递策略；最后，系统面向具体 JD 生成可执行的简历优化建议，用户确认采纳后由 AI 生成改写草稿，经差异预览确认后保存为新的简历版本。

在工程架构上，后端采用模块化单体（Modular Monolith）设计，按领域纵切为认证、画像、简历、岗位、匹配、推荐、报告、改写等独立模块，模块之间仅通过 service 接口协作、不直接引用彼此的数据访问层，从而在保证交付速度的同时预留了未来拆分为微服务的清晰边界。前端采用工作台式布局，AI 助手以不同角色（求职引导员、岗位分析师、简历编辑顾问、投递策略顾问）贯穿全流程，而非仅提供一个聊天入口。

## 三、AI 工具选型理由

在 AI 能力的工程接入上，我们设计了一套 Provider 适配层：通过 `LLMProvider` 抽象基类统一定义 `complete()` 和 `embed()` 接口，具体实现可以是 OpenAI 兼容 API、DeepSeek、通义千问或智谱等任意供应商，切换时只需更改配置而无需改动业务代码，从根本上避免了厂商绑定。为了让开发和持续集成不依赖外部服务，系统默认使用确定性的 Fake Provider（`AI_PROVIDER=fake`），它返回结构完整但内容固定的响应，使得全部测试可以在无 API Key 的环境下端到端跑通。向量检索方面，MVP 阶段选择 pgvector 作为 PostgreSQL 扩展而非引入独立向量数据库，以降低运维复杂度并与主库统一管理。匹配评分采用规则引擎与大语言模型混合的策略：技能重叠度、关键词覆盖率等可确定性计算的维度交由规则引擎处理以保证结果可复现，而语义层面的理解和解释则由 LLM 补充完成。Embedding 维度统一为 256 维，通过标准化接口封装，底层模型可按需替换。这套设计使得测试不依赖外部服务、供应商可随时切换、评分结果既可解释又部分可复现。

## 四、关键配置

整个项目采用 Monorepo 组织，前端（`apps/web`）、后端（`services/api`）、基础设施（`infra`）和文档（`docs`）共处同一仓库，便于原子提交和前后端契约共享。后端技术栈为 Python 3.12 + FastAPI + SQLAlchemy 2（异步模式）+ Alembic 数据库迁移 + PostgreSQL/pgvector + Redis + Arq 异步任务队列。前端技术栈为 React 18 + Vite 6 + TypeScript + TanStack Query + Tailwind CSS + shadcn/ui 风格组件。环境隔离策略上，开发机使用 conda 环境（`offerpilot`）隔离 Python 解释器、nvm 锁定 Node 22，通过 `make setup` 一键搭建；部署机则使用 Docker + compose 构建不可变镜像，确保环境可移植。前后端之间通过统一的响应信封（`{data, requestId, timestamp}`）和集中式错误码体系进行契约约定，保证接口一致性和可观测性。前端静态资源可部署到 GitHub Pages 或 Nginx，后端以 Docker 镜像独立部署。
